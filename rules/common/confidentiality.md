# Confidentiality Rules

## Always

- Create drafts for all outbound communications — never auto-send
- Flag when a document contains potentially sensitive information before presenting it
- Redact or generalize financial figures in summaries unless explicitly requested to include them
- Ask for confirmation before including specific numbers in external-facing documents

## Never

- Include internal project codenames in external communications
- Share employee personal information (salary, performance ratings, personal contact info)
- Include unreleased product details or roadmap items in external documents
- Expose internal tool URLs, API keys, credentials, or system architecture details
- Share customer-specific data across different customer contexts
- Forward internal Slack messages or email threads to external parties without review

## When Uncertain

- Default to more restrictive handling
- Flag the item and ask the user: "This appears to contain [type] information. Is it appropriate to include in this [context]?"
- Recommend legal review for anything that could have regulatory implications

## Data Classification

| Level | Examples | Handling |
|-------|---------|---------|
| **Public** | Published content, marketing materials | Share freely |
| **Internal** | Team updates, project status, org charts | Internal only |
| **Confidential** | Financial data, strategies, customer data | Need-to-know only |
| **Restricted** | Credentials, legal matters, personal data | Explicit authorization required |
